# 01. Problem Statement

## 1. Операционная проблема

Ручное редактирование `/etc/nginx/*.conf` приводит к системным рискам:

- отсутствует единый источник истины;
- изменения не атомарны;
- rollback нестандартизирован;
- drift между ожидаемой и реальной конфигурацией накапливается;
- аудит изменений затруднён.

## 2. Цели решения

1. Сделать `state.json` единственным управляемым состоянием.
2. Перевести `/etc/nginx` в роль runtime output.
3. Ввести транзакционный apply и гарантированный rollback.
4. Формализовать разделение control plane и data plane.

## 3. Функциональные требования (FR)

- **FR-1**: система должна хранить декларативное состояние в `/etc/nginx-admin/state.json`.
- **FR-2**: runtime-конфиги должны генерироваться только из `state.json`.
- **FR-3**: apply должен выполняться через стадию staging во временной директории.
- **FR-4**: перед commit обязательно выполняется `nginx -t`.
- **FR-5**: перед commit обязательно создаётся backup (`/etc/nginx` + `state.json`).
- **FR-6**: commit должен быть атомарным относительно runtime-конфига.
- **FR-7**: после commit выполняется `nginx reload`.
- **FR-8**: при любой ошибке до commit runtime не должен изменяться.
- **FR-9**: система должна поддерживать режимы Overlay и Full Control.
- **FR-10**: должна существовать операция rollback к конкретному backup.
- **FR-11**: система должна обнаруживать и устранять drift.
- **FR-12**: API должен поддерживать optimistic concurrency через revision/etag.
- **FR-13**: apply-операция должна быть идемпотентной для идентичного state.
- **FR-14**: каждая apply-операция должна иметь аудитный trace id и статус.

## 4. Ограничения

- NGINX остаётся единственным data plane runtime.
- Источник истины на хосте локальный файл (без внешнего KV-store в базовой версии).
- Коммит конфигурации выполняется на одном хосте атомарными файловыми операциями.

## 5. Критерий успеха

Операционный процесс изменения конфигурации NGINX полностью проходит через state -> apply pipeline, без ручного редактирования runtime-файлов.
